# 4. Casos de Teste

## Fatias modeladas — 2 casos por fatia, padrão IEEE-829

> **Critério de seleção**: para cada fatia, 1 caso de "caminho feliz" e 1 caso de **fronteira** (BVA, transição crítica de estado ou caminho de erro). Casos derivados diretamente dos diagramas comportamentais (Seções 3.1, 3.2, 3.3).

---

## 4.1 Casos de Teste — Fatia 1

### Cliente realiza pedido com pagamento e alocação automática de entregador

---

### TC-FATIA1-01 — Pedido concluído com sucesso (caminho feliz)

| Campo                           | Conteúdo                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                          | TC-FATIA1-01                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Fatia / História**            | Fatia 1 — US-PAG-001 (pagar com segurança) + US-LOG-001 (alocar entregador automaticamente)                                                                                                                                                                                                                                                                                                                                                                                        |
| **Pré-condições**               | Cliente autenticado com CPF válido; carrinho com 2 itens (Hambúrguer R$ 28,00 + Batata R$ 12,00 = R$ 40,00); taxa de entrega R$ 8,00; cartão de crédito cadastrado e válido; ao menos 1 entregador disponível a menos de 5km da empresa parceira; empresa parceira com `aceita_pedidos = true`.                                                                                                                                                                                    |
| **Dados de entrada**            | `pedido_id` gerado pelo sistema; `forma_pagamento = CARTAO_CREDITO`; `cartao_token = tok_visa_valid_4242`; sem cupom aplicado; `endereco_entrega_id` = endereço principal do cliente.                                                                                                                                                                                                                                                                                              |
| **Passos**                      | 1) Cliente clica em "Finalizar Pedido"; 2) Backend valida disponibilidade de todos os itens; 3) Backend cria Pedido (status: `AGUARDANDO_PAGAMENTO`); 4) Cliente clica em "Pagar"; 5) Backend envia requisição ao GatewayPagamento; 6) Gateway retorna `status: APROVADO` + `transacao_id`; 7) Backend atualiza Pedido para `PAGO`; 8) ServicoMatching encontra entregador; 9) Entregador recebe notificação e aceita em 45 segundos; 10) Backend cria Entrega (estado: `ACEITA`). |
| **Resultado esperado**          | Sistema retorna HTTP 200 com `{ pedido_id, status: "PAGO", entregador: { nome, placa, avaliacao_media }, tempo_estimado_minutos }`; cliente recebe push notification "Pagamento confirmado! João S. está a caminho de buscar seu pedido."; empresa parceira recebe push "Novo pedido #{numero} — prepare para entrega."; estado da Entrega = `ACEITA`; `ENTREGADOR.disponivel = false`.                                                                                            |
| **Critério de aprovação**       | a. `PEDIDO.status = 'PAGO'` no banco; b. `PAGAMENTO.status = 'APROVADO'` com `transacao_gateway_id` preenchido; c. `ENTREGA.estado = 'ACEITA'` criada com FK para pedido e entregador; d. `ENTREGADOR.disponivel = false`; e. Tempo total do fluxo (steps 1-10) < 8 segundos.                                                                                                                                                                                                      |
| **Severidade em caso de falha** | **Crítica** — é o fluxo principal do produto; falha impede qualquer receita.                                                                                                                                                                                                                                                                                                                                                                                                       |

### TC-FATIA1-02 — Pagamento recusado: reserva de estoque deve ser revertida (caso de erro crítico)

| Campo                           | Conteúdo                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                          | TC-FATIA1-02                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Fatia / História**            | Fatia 1 — US-PAG-001 (pagamento seguro) — caminho de erro                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Pré-condições**               | Cliente autenticado; carrinho com 1 item (Refrigerante R$ 6,00, `quantidade_estoque = 3`); taxa de entrega R$ 5,00; gateway configurado em modo mock para retornar `INSUFFICIENT_FUNDS`; estoque do item = 3 unidades antes do teste.                                                                                                                                                                                                                                               |
| **Dados de entrada**            | `forma_pagamento = CARTAO_CREDITO`; `cartao_token = tok_visa_declined_insufficient`; quantidade solicitada = 1 unidade.                                                                                                                                                                                                                                                                                                                                                             |
| **Passos**                      | 1) Cliente finaliza pedido; 2) Backend cria Pedido (`AGUARDANDO_PAGAMENTO`) e aplica soft lock no estoque (`quantidade_reservada++`); 3) Cliente clica "Pagar"; 4) Gateway retorna `402 Payment Required { code: "INSUFFICIENT_FUNDS" }`; 5) Backend recebe resposta de recusa; 6) Backend libera soft lock de estoque; 7) Backend atualiza Pedido para `PAGAMENTO_RECUSADO`; 8) Backend registra Pagamento com `status = RECUSADO`; 9) Frontend exibe mensagem de erro ao cliente. |
| **Resultado esperado**          | Frontend exibe: "Pagamento recusado: saldo insuficiente. Verifique seu cartão ou escolha outra forma de pagamento."; carrinho permanece intacto com os mesmos itens; `PRODUTO.quantidade_estoque = 3` (sem desconto); nenhuma Entrega criada; nenhum entregador notificado.                                                                                                                                                                                                         |
| **Critério de aprovação**       | a. `PEDIDO.status = 'PAGAMENTO_RECUSADO'`; b. `PAGAMENTO.status = 'RECUSADO'` com `transacao_gateway_id = null`; c. `PRODUTO.quantidade_estoque` = 3 (valor original, sem desconto); d. Nenhuma linha em `ENTREGA` com esse `pedido_id`; e. Mensagem de erro exibida é específica (menciona "saldo insuficiente", não erro genérico).                                                                                                                                               |
| **Severidade em caso de falha** | **Crítica** — se estoque não for revertido, produto fica indisponível indevidamente; se entregador for notificado sem pagamento, há prejuízo operacional.                                                                                                                                                                                                                                                                                                                           |

## 4.2 Casos de Teste — Fatia 2

### Ciclo de vida da entrega com rastreamento em tempo real

---

### TC-FATIA2-01 — Transição de estado AguardandoAceite → Cancelada por timeout (caso de fronteira temporal)

| Campo                           | Conteúdo                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                          | TC-FATIA2-01                                                                                                                                                                                                                                                                                                                                                                      |
| **Fatia / História**            | Fatia 2 — US-LOG-003 (visualizar e aceitar entrega) — transição crítica por timeout                                                                                                                                                                                                                                                                                               |
| **Pré-condições**               | Pedido `PAGO`; Entrega criada no estado `AguardandoAceite`; 3 entregadores disponíveis na região, todos configurados em mock para **não responder** às notificações; sistema com timer de timeout configurado para 2 minutos por rodada, máximo 3 rodadas.                                                                                                                        |
| **Dados de entrada**            | `entrega_id = ENT-0042`; `entregadores_disponiveis = [ENT001, ENT002, ENT003]`; todos com mock `resposta = null` (sem aceite nem recusa); clock de teste acelerado para simular 10 minutos.                                                                                                                                                                                       |
| **Passos**                      | 1) Sistema envia push para ENT001, ENT002, ENT003 (rodada 1); 2) Aguardar 2min — nenhum aceita; 3) Sistema busca próximos 3 entregadores (rodada 2); 4) Aguardar 2min — nenhum aceita; 5) Sistema busca próximos 3 entregadores (rodada 3); 6) Aguardar 2min — nenhum aceita; 7) Sistema esgota tentativas.                                                                       |
| **Resultado esperado**          | Após 3 rodadas sem aceite: `ENTREGA.estado = 'AGUARDANDO_ENTREGADOR'`; cliente recebe push: "Seu pedido foi confirmado, mas estamos procurando um entregador disponível. Você será notificado assim que atribuirmos."; alerta enviado para equipe operacional (log ou fila de ops); **pagamento NÃO é estornado automaticamente** neste momento (cliente aguarda).                |
| **Critério de aprovação**       | a. `ENTREGA.estado = 'AGUARDANDO_ENTREGADOR'` após exatamente 3 rodadas de timeout; b. Log de alerta operacional registrado com `entrega_id` e timestamp; c. `PEDIDO.status` permanece `'PAGO'` (não cancelado); d. Push de aviso enviado ao cliente (verificar fila de notificações); e. `ENTREGADOR.disponivel` de todos os 3 mantém `true` (nenhum foi bloqueado sem aceitar). |
| **Severidade em caso de falha** | **Alta** — se a entrega ficar presa em `AguardandoAceite` para sempre, nenhum mecanismo de recuperação é ativado; cliente fica sem feedback.                                                                                                                                                                                                                                      |

## 4.3 Casos de Teste — Fatia 3

### Sistema de gamificação e bonificação para entregadores

---

### TC-FATIA3-01 — Cálculo de bonificação com múltiplos multiplicadores ativos (caso de fronteira composto)

| Campo                           | Conteúdo                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                          | TC-FATIA3-01                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Fatia / História**            | Fatia 3 — US-GAM-005 (calcular bonificação variável) + US-BON-002 (bonificação em horário de pico)                                                                                                                                                                                                                                                                                                                                                                     |
| **Pré-condições**               | Entregador de nível OURO (`percentualBase = 9%`); entrega concluída com `valorFrete = R$ 15,00`; horário de conclusão = 19h30 (dentro do pico 18h-22h); entrega realizada no bairro "Pinheiros" (região de alta demanda, `multiplicador = 1.20` na `TABELA_MULTIPLICADORES`); `streakDias = 7` (streak ativo de 7 dias).                                                                                                                                               |
| **Dados de entrada**            | `entregador_id = ENT-501`; `entrega_id = ENT-7001`; `valor_frete = 15.00`; `timestamp_entrega = 2026-05-24T19:30:00`; `bairro_destino = "Pinheiros"`.                                                                                                                                                                                                                                                                                                                  |
| **Passos**                      | 1) Sistema recebe evento de entrega confirmada; 2) Sistema recupera `PerfilGamificacao` do entregador; 3) Sistema aplica `percentualBase = 9%` (nível OURO); 4) Sistema calcula `bonificacaoBase = 15.00 × 0.09 = R$ 1,35`; 5) Sistema aplica `multiplicadorHorario = 1.30` (19h30 = pico); 6) Sistema aplica `multiplicadorRegiao = 1.20` (Pinheiros = alta demanda); 7) Sistema aplica `multiplicadorStreak = 1.20` (7+ dias); 8) Sistema calcula bonificação final. |
| **Resultado esperado**          | `bonificacaoFinal = 1.35 × 1.30 × 1.20 × 1.20 = R$ 2,53`; `PERFIL_GAMIFICACAO.pontos_acumulados` incrementado conforme fórmula de pontos; histórico de bonificação registrado com breakdown dos multiplicadores; entregador notificado: "Você ganhou R$ 2,53 de bonificação nesta entrega! 🎉".                                                                                                                                                                        |
| **Critério de aprovação**       | a. `bonificacaoFinal = 2.53` (arredondado para 2 casas decimais, tolerância ±R$0.01); b. Histórico registrado com os 4 multiplicadores individuais (auditável); c. Push de notificação enviado com valor correto; d. `pontos_acumulados` incrementado pelo valor correto.                                                                                                                                                                                              |
| **Severidade em caso de falha** | **Alta** — erro de cálculo de bonificação gera reclamações financeiras dos entregadores e pode virar passivo trabalhista.                                                                                                                                                                                                                                                                                                                                              |

### TC-FATIA3-02 — Desbloqueio de badge no limiar exato de 100 entregas (análise de valor limite — BVA)

| Campo                           | Conteúdo                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                          | TC-FATIA3-02                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Fatia / História**            | Fatia 3 — US-GAM-003 (desbloquear badges por conquistas) — Boundary Value Analysis                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Pré-condições**               | Entregador com `total_entregas = 99` antes desta entrega; badge `CENTENARIO` com critério `100_ENTREGAS` existe no banco e **não** está em `ENTREGADOR_BADGE` para este entregador; badge `CINQUENTA_ENTREGAS` já desbloqueado (para confirmar idempotência).                                                                                                                                                                                                                                            |
| **Dados de entrada**            | `entregador_id = ENT-202`; entrega `ENT-8000` confirmada — esta será a 100ª entrega; `total_entregas atual = 99`.                                                                                                                                                                                                                                                                                                                                                                                        |
| **Passos**                      | 1) Entrega ENT-8000 confirmada; 2) Sistema atualiza `total_entregas = 100`; 3) Sistema executa verificação de badges (`Badge.verificarDesbloqueio()`); 4) Sistema verifica critério `100_ENTREGAS` — `total_entregas >= 100` → **verdadeiro**; 5) Sistema insere em `ENTREGADOR_BADGE` com `data_desbloqueio = now()`; 6) Sistema notifica entregador.                                                                                                                                                   |
| **Resultado esperado**          | Badge `CENTENARIO` desbloqueado; `ENTREGADOR_BADGE` contém nova linha `(ENT-202, badge_centenario, timestamp)`; push enviado: "🏆 Conquista desbloqueada: CENTENÁRIO! Você completou 100 entregas!"; badge `CINQUENTA_ENTREGAS` **não** é duplicado (idempotência verificada).                                                                                                                                                                                                                           |
| **Critério de aprovação**       | a. Exatamente 1 nova linha em `ENTREGADOR_BADGE` para o entregador (apenas `CENTENARIO`); b. `data_desbloqueio` preenchida com timestamp do momento da entrega; c. Push enviado com nome correto do badge; d. Badge `CINQUENTA_ENTREGAS` não duplicado — verificar `COUNT(*) WHERE entregador_id = ENT-202 AND badge_id = badge_50 = 1`; e. **Teste negativo complementar**: com `total_entregas = 99` (antes), nenhum badge deveria ser desbloqueado — verificar que o limiar é exatamente 100, não 99. |
| **Severidade em caso de falha** | **Média** — falha não impede operação, mas afeta engajamento e confiança dos entregadores no sistema de gamificação.                                                                                                                                                                                                                                                                                                                                                                                     |
